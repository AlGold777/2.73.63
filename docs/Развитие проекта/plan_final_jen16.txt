**Финальный подробный план (jen16) — документ для реализации LLM**

Этот документ должен быть достаточно конкретным, чтобы даже слабая модель не пошла «не туда».

---

**Цель**
Стабилизировать расширение (безопасность, память, UX, надежность), без крупных миграций (TS/Vite/Hybrid).

**Не входит в текущий цикл**
- TypeScript миграция
- Vite/Webpack build pipeline
- Hybrid (API + Web)

**Важные правила окружения**
- MV3 service worker может "уснуть" → не полагаться на `setInterval` для критических задач.
- Не использовать ES-модули в background, если `manifest.json` не переведён на `"type": "module"`.
- Любые новые файлы в content scripts должны быть добавлены в `manifest.json` **в правильном порядке** (порядок важен).
- Любые новые файлы в background должны быть подключены через `importScripts()` **или** через переход на `type: module` (выбрать один путь, не смешивать).

---

**Критические уточнения (Patch v1) — обязательно к реализации**

**Зачем:** закрыть пробелы по race conditions, attach‑таймингам и недоописанным контрактам.

1) **Race Conditions / Handshake‑протокол**
- Добавить явный handshake между background и content script:
  - Content → `SCRIPT_READY` (с `tabId`, `llmName`, `tabSessionId`).
  - Background → `ACK_READY` (с `tabSessionId`, `dispatchId`).
  - Любой dispatch отправляется **только после** `ACK_READY`.
- Обработать случай, когда `SCRIPT_READY` пришёл до listener:
  - Content повторяет `SCRIPT_READY` каждые 2s до `ACK_READY`, но не более 5 попыток.
- Реакция на SPA‑redirect:
  - Content отправляет повторный `SCRIPT_READY` при `popstate` и при изменении `location.href`.
- Критерий: нет dispatch без ACK; тайм‑аут ожидания ACK = 6s.

2) **AttachmentHandler — ожидание завершения загрузки**
- `attach()` возвращает `{ success, uploadedCount, failedFiles, reason }`.
- После attach ждать платформенный индикатор:
  - Claude: появление атрибута `data-upload-id` или исчезновение элемента загрузки.
  - Gemini: появление thumbnail/preview элемента.
  - Perplexity: появление чипа файла или завершение progress‑bar.
  - Qwen: появление мини‑карточки/иконки файла рядом с композером.
- Тайм‑аут на файл: 10s, после чего `reason = 'TIMEOUT'`.
- Критерий: `send()` вызывается только после подтверждения загрузки.

3) **SafeMutex — точный контракт**
- API: `runExclusive(fn, { timeoutMs = 30000, queueLimit = 10 })`.
- FIFO очередь; при превышении `queueLimit` → reject `QueueFullError`.
- `timeoutMs` → reject `TimeoutError`, но `fn` продолжает выполняться.
- `finally` всегда освобождает lock и запускает следующую задачу.
- Критерий: deadlock невозможен, ошибки всегда пробрасываются.

4) **TTLMap — точные правила**
- `touchOnGet = true` (обновлять timestamp на каждом get).
- Очистка oldest‑first, только expired‑entries (валидные не трогать).
- `_lazyCleanup(limit=10)` не запускается при `get()` на несуществующие ключи.
- Критерий: активно используемые ключи не вычищаются преждевременно.

5) **Remote selectors — доверие только к коду**
- `TRUSTED_HASH` захардкожен в коде, не хранится в storage.
- Обновление hash — только через релиз расширения.
- При несовпадении: fallback на встроенные селекторы + warning‑лог.
- Критерий: MITM не может подменить селекторы без релиза.

6) **SMART‑критерии (обязательные измеримые цели)**
- Инкрементальный рендер: обновление 100 элементов < 16ms; изменение 1 элемента → 1 DOM update.
- Focus минимизация: максимум 2 tab‑switch на dispatch; если `requiresFocus=false` → 0.
- Cleanup: после закрытия таба все ключи `tabId` удалены; таймеры очищены; jobState без записи.

7) **Минимальные acceptance‑проверки**
- `rg -n "innerHTML\\s*="` → только через `sanitizeHTML`.
- Сценарий: dispatch → close tab → не остаётся state в `chrome.storage.session`.
- Сценарий: attach файлы в Claude/Gemini → отправка только после upload.
- Сценарий: повторный `SCRIPT_READY` → ACK получен → dispatch успешен.

**🔴 Критично**

1) **DOMPurify для `innerHTML`**
- **Где:** `results.js`, `results-shared.js`, все content scripts, любые прямые `innerHTML = ...`.
- **Добавить файл:** `utils/sanitize.js`.
- **Подключение:**
  - В content scripts: добавить DOMPurify в `manifest.json` перед скриптами, которые используют `sanitizeHTML`.
  - В `result_new.html` (или где рендерится results UI) тоже подключить DOMPurify.
- **Реализация `sanitizeHTML` (строго):**
  - Ограничить теги: `b, i, em, strong, p, br, ul, ol, li, code, pre, a`.
  - Разрешить атрибуты: `href, class`.
  - Запретить `javascript:` и `data:` ссылки (через `ALLOWED_URI_REGEXP`).
  - Для ссылок: автоматически выставлять `rel="noopener noreferrer"`.
- **Требование:** никакого прямого `innerHTML = ...` без `sanitizeHTML`.
- **Критерий готовности:** проверка `rg -n "innerHTML\s*="` → все присваивания заворачиваются через `sanitizeHTML`.

2) **TTLMap — выборочно, без `setInterval`**
- **Использовать TTLMap только для:**
  - `platformRunHistory`
  - `selectorResolutionMetrics`
  - `telemetrySampleCache`
- **Не использовать TTLMap для:**
  - `promptDispatchMutexByLlm`
  - `promptSubmitWaiters`
  - `rateLimitTimers`
  Эти структуры очищаются явно при завершении операций.
- **Добавить файл:** `utils/ttl-map.js`.
- **Реализация TTLMap:**
  - Ленивая очистка **в `get()` и `set()`**.
  - Чистить не более 10 записей за операцию, чтобы не блокировать.
  - Не использовать `setInterval`.
  - По желанию обновлять timestamp на `get` (LRU-подобное поведение) — явно описать в коде.
- **Критерий готовности:** Map не растут бесконечно; активные операции не ломаются из-за TTL.

3) **SafeMutex с пробросом ошибок**
- **Где:** `background.js` (замена `withPromptDispatchLock`).
- **Добавить файл:** `utils/safe-mutex.js`.
- **Требование:** ошибки не глотать. Логировать и `throw`.
- **Критерий готовности:** caller получает ошибку при сбое dispatch.

4) **Cleanup при закрытии вкладок / уходе с домена**
- **Где:** `background.js`.
- **Что:** единая функция `cleanupTabResources(tabId)`.
- **Внутри очистить:**
  - `pingWindowByTabId`
  - `promptDispatchMutexByLlm`
  - `promptSubmitWaiters`
  - `rateLimitState`
  - `rateLimitTimers`
  - `jobState.llms[llmName]`
  - `TabMapManager` (removeByTabId)
  - отмена deferred timers (если есть)
- **Хуки:**
  - `chrome.tabs.onRemoved`
  - `chrome.tabs.onUpdated` → только если `changeInfo.url` и вкладка уходит с нужного домена
- **Критерий готовности:** закрытие вкладки освобождает все связанные структуры.

5) **Storage Compression (LZString) + миграция + pruning**
- **Добавить файл:** `utils/storage-compress.js`.
- **Подключение LZString:** `lib/lz-string.min.js`, подключить в background через `importScripts()`.
- **Обязательные исправления:**
  - `chrome.storage.local.getBytesInUse(null)` возвращает число.
  - `QUOTA_BYTES` может быть `undefined` → fallback `10 * 1024 * 1024`.
  - Для несжатых данных хранить **оригинальный объект**, не JSON-строку.
- **Алгоритм хранения:**
  - Если payload < 10KB → сохранять как объект.
  - Если payload >= 10KB → сжать `LZString.compressToUTF16` и сохранить как строку с префиксом `__LZ__`.
  - Если сжатие не дало выигрыша (например < 10%) → сохранить как объект.
- **Алгоритм чтения:**
  - Если строка и начинается с `__LZ__` → распаковать и `JSON.parse`.
  - Если строка без префикса → `JSON.parse` (для миграции старого формата).
  - Если объект → вернуть как есть.
- **Миграция:** однократная по флагу `__storage_migrated_v2`.
- **Pruning:** если использование >80% → удалить старые логи/результаты (старше 7 дней).
- **Критерий готовности:** большие payload’ы не переполняют storage; старые данные корректно читаются.

6) **Инкрементальный рендер `results.js`**
- **Цель:** убрать полный `innerHTML = ''` и мигание.
- **Реализация:**

7) **AttachmentHandler (per-platform + явный fallback)**
- **Добавить файл:** `content-scripts/attachment-handler.js`.
- **Стратегии:**

- **Fallback:** если стратегия не сработала, вернуть `{ success: false, reason: 'MANUAL_REQUIRED', message: ... }`.
- **Критерий:** там, где автоматизация невозможна, пользователь явно видит инструкцию.

**🟢 Архитектурные улучшения**

13) **Split `background.js` на модули**
- **Важно:** если не переводить background в `type: module`, использовать `importScripts()`.
- **Целевая структура:**
  - `background/index.js` — entry (≤500 строк, только wiring/initialization).
  - `background/tab-manager.js` — управление `TabMapManager`, lifecycle hooks, cleanup hooks.
  - `background/dispatch-coordinator.js` — core `dispatchPromptToTab`, retries, focus coordination, telemetry emits.
  - `background/job-orchestrator.js` — `startModelForLLM`, `runModelThroughTabs`, job state persistence, human visit orchestration.
  - `background/message-router.js` — `chrome.runtime.onMessage`, `sendMessageSafely`, adapters for telemetry/events.
- **Имплементация:**
  1. `index.js` подключает `importScripts()` с новыми модулями, регистрирует слушатели (`tabs`, `alarms`, `runtime`, `storage`), и стартует `ensureInitialState`.
  2. Каждый модуль экспортирует namespace/функции, которые `index.js` использует (например, `dispatchCoordinator.dispatchPromptToTab`).
  3. Перенести существующие глобальные функции/переменные в соответствующие модули с минимальными пересечениями; shared state (`jobState`, `TabMapManager`, `selectorResolutionMetrics`) либо делится через getter/setter или singletons.
  4. Убедиться, что порядок импорта через `importScripts` сохраняет зависимости (`tab-manager` до `dispatch-coordinator`, поскольку последний читает `TabMapManager`).
- **Acceptance:**
  - `background/index.js` ≤ 500 строк, содержит только wiring + feature toggles.
  - Модули покрыты unit/acceptance tests (если есть) и документируются (README в `background/`).

14) **Semantic selectors fallback**
- ARIA/text/placeholder fallback, генерация CSS selector и кеширование.
- **Критерий:** селекторы восстанавливаются без ручного вмешательства.

15) **State Machine для dispatch lifecycle**
- Ввести явные состояния и переходы.
- **Критерий:** нет разрозненных флагов dispatch.

16) **Event-driven SCRIPT_READY**
- Content script сам отправляет `SCRIPT_READY`.
- Background не делает polling.

17) **Circuit Breaker persistence**
- Хранение в `chrome.storage.session`, восстановление при старте background.

18) **Retry + Circuit Breaker (идемпотентность, классификация)**
- **Примечание:** архитектурный раздел, перенесено из high-priority; требует глобальной координации dispatcher + utilities.
- **Добавить файл:** `utils/retry-strategy.js` (если ещё не готов) и связать с background.
- **Классы ошибок:**
  - `PERMANENT` (AUTH, FORBIDDEN, INVALID_KEY) → не ретраить.
  - `RATE_LIMITED` → ретрай с повышенной задержкой.
  - `TRANSIENT` (timeout/network) → обычный backoff.
- **Идемпотентность:**
  - `dispatchId` добавляется в каждую отправку, `DispatchIdRegistry` хранит timestamps (TTL 60s).
  - До повторной отправки проверять, не был ли тот же `dispatchId` недавно «подтверждён».
- **Circuit Breaker:**
  - failureThreshold=5, recoveryTime=60s.
  - persist state с дебаунсом ~2s в `chrome.storage.session` (или `local` как fallback).
- **Пошагово:**
  1. `dispatchId` генерируется в `dispatchPromptToTab`, записывается в реестр, логируется.
  2. Ошибки классифицируются через `RetryStrategy.classifyError`, `shouldRetry`/`getDelay` и ограничиваются `maxAttempts`.
  3. `CircuitBreaker.recordFailure/recordSuccess` переключает состояния, переходит в `OPEN` при 5 провалах, восстанавливается через 60s.
  4. При попытке dispatch, когдаBreaker открыт, выдать `updateModelState` + telemetry и не запускать новую сессию.
- **Acceptance:**
  - `maxAttempts` не превышается (обычно ≤5).
  - `chrome.storage.session` содержит `circuitBreakerState` только после debounce.
  - `PROMPT_SUBMITTED` с повторным `dispatchId` внутри 60s игнорируется (лог `PROMPT_SUBMITTED_STALE`).
  - Rate-limit errors увеличивают задержку, permanent errors прерывают retries.
- **Критерий:** ретраи конечные, дубли не отправляются.

19) **Focus минимизация**
- **Ключевая правка:** `chrome.tabs.sendMessage` не принимает `timeout` → использовать `Promise.race`.
- **Поток:**
  1) `GET_ANSWER_NO_FOCUS` с `sendMessageWithTimeout`.
  2) При `requiresFocus`/таймауте — минимальный `withPromptDispatchFocusLock`: активировать вкладку, вызвать `GET_ANSWER`, дождаться `PROMPT_SUBMITTED`, вернуть фокус.
  3) Учитывать `focusSwitches` и сохранять в `jobState`.
- **Acceptance:**
  - Если `requiresFocus=false`, `focusSwitches === 0`.
  - `Promise.race` тайм-аутит через 5s без переключений.
  - После диспатча с фокусом предыдущая вкладка восстанавливается <200ms.
  - Логи содержат `focusSwitches` count (max 2).
- **Критерий:** лишние переключения минимальны.

20) **Selector cache cleanup через background alarm**
- **Важно:** `chrome.alarms` работает только в background; content scripts проверяют TTL, но не чистят сами.
- **Пошагово:**
  1. Добавить `SELECTOR_CACHE_CLEANUP_ALARM` (10 минут), запускать на старте/`onStartup`.
  2. `cleanupExpiredSelectorCache` сравнивает `timestamp`, удаляет `selector_cache_*` записи старше 30 дней (не более 100 за прогон).
  3. Content scripts просто читают запись и проверяют TTL (без cleanup). Если запись просрочена, возвращают `null`.
  4. Background логирует удалённые ключи для диагностики.
- **Acceptance:**
  - Alarm срабатывает каждые 10 минут (см. `chrome.alarms`).
  - `readCache` не вызывает `cleanupExpiredCache`.
  - `cleanupExpiredSelectorCache` удаляет только устаревшие ключи и не обрабатывает более 100 записей за прогон.
- **Критерий:** фоновый cleanup не блокирует чтения.
