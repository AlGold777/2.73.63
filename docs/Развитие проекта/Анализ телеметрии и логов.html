Ниже — разбор по факту ваших файлов telemetry-.json и All Logs.md: что в текущей телеметрии/логах мало полезно / вредно для анализа, и чего критически не хватает, чтобы быстро и однозначно объяснять “почему completion не определился без user-focus”.

1) Что сейчас “шум” или снижает ценность данных
1.1 Нестандартизированные label (вредно)

В телеметрии есть события, где label — это текст шага:

"Typing start", "Prompt injected", "Composer found", "Attempt via Ctrl+Enter" и т.д.

Это ломает агрегацию (label-space раздувается), затрудняет статистику и корреляцию.

Как должно быть: один стабильный label типа PIPELINE_STEP, а название шага — в meta.step.

1.2 Дубли PIPELINE_COMPLETE / неполные метаданные

PIPELINE_COMPLETE встречается дважды для одной модели с разным набором полей:

в одном варианте нет tabId/extVersion, в другом — есть; при этом requestId часто None, completionReason: "unknown", answerLength может быть подозрительно малым (например 15).

Это делает итоговый статус неаудируемым: непонятно “какой complete считать истинным”.

1.3 Неконсистентный sessionId (критично для корреляции)

В Round events sessionId выглядит как run session (число 1769805454170),

а в PIPELINE_ERROR/PIPELINE_COMPLETE — как tabSession/provider session (chatgpt_mit…, qwen_mit…).

В итоге один и тот же ключ означает разное, а join между событиями становится ненадёжным.

1.4 requestId часто None именно в важных местах

В ROUND1_START requestId есть, а в PIPELINE_ERROR / PIPELINE_COMPLETE — часто None.

То есть вы теряете связку “этот error/complete относится к этому dispatch”.

1.5 SELECTOR_MISS-спам без агрегации

SELECTOR_MISS доминирует по частоте. Сами по себе промахи важны, но поштучная запись каждого miss:

-

раздувает лог,

-

не даёт понять “это нормально/ожидаемо или регресс” без контекста,

-

не говорит “а сколько было HIT”.

Лучше: агрегировать “miss count per target/phase/window” + периодические сэмплы.

1.6 TAB_VISIT в телеметрии очень много (особенно после user-focus)

В 21:46 прогоне TAB_VISIT резко больше (видно human_visit, user_focus). Это полезно для RCA, но в текущем виде:

-

не видно, что именно изменилось (DOM? рост текста? смена кнопок?) после визита,

-

а значит визиты превращаются в “шум про переключения”.

2) Чего критически НЕ хватает (чтобы RCA делался за минуты)
2.1 Единые корреляционные ключи во ВСЕХ событиях (обязательное)

Нужно, чтобы каждое событие несло одинаковый набор:

-

runSessionId (глобальный запуск)

-

dispatchId (на конкретную модель)

-

tabId

-

tabSessionId (если используете)

-

llmName

Сейчас это размазано: round-и живут в одном мире, pipeline — в другом.

2.2 Нет Round2 (VERIFY) событий вообще

В обоих telemetry JSON 0 событий ROUND2.

Это делает невозможным понять: verify действительно отключён/не исполняется, или исполняется, но не логируется.

2.3 Нет явной FSM / state transitions по модели

Нужны события вида:

-

STATE_CHANGE: DISPATCHING → GENERATING → FINALIZING → COLLECTING → DONE_*

Сейчас вы пытаетесь восстановить состояния косвенно (по selector miss, streaming_done и т.п.), это плохо масштабируется.

2.4 Нет “completion detector trace”

Чтобы понять “почему не определили конец”, нужны данные детектора:

-

какой сигнал использовался (ui / stabilization / provider_flag)

-

doneConfidence

-

lastMessageHash/len + delta (пусть агрегировано)

-

сколько “stable checks” было

-

какая кнопка видна (stop/send/regenerate)

-

document.hidden, hasFocus (в момент проверок)

Сейчас есть только visible:false на selector miss, но нет динамики “росло/стабилизировалось”.

2.5 Нет событий handshake/портов на критическом пути

Вы упоминали The message port closed... — но в телеметрии это не видно как структурированное событие с:

-

tabId, llmName, phase, attempt, recoverAction.

Без этого recovery невозможно оценивать.

2.6 Нет инцидент-снимка для MODEL_MISSING / integrity

Если появляется MODEL_MISSING или activeModels=[], нужно логировать snapshot:

-

список bindings (llm→tab)

-

активные модели jobState

-

текущая round/phase

-

таймеры/ожидания (кратко)

Иначе “почему пропали модели” не расследуется.

2.7 Нет “selector HIT rate” и версий селектор-пакета

Одни miss-ы не дают качества. Нужны:

-

SELECTOR_HIT (хотя бы счётчики)

-

selectorPackVersion / providerProfileVersion (чтобы видеть регресс после обновления)

3) Минимальный набор изменений (наиболее выгодный)

1.

Стабилизировать schema: фиксированный label, meta.step, единые IDs.

2.

Добавить ROUND2_START/END и verify outcome.

3.

Добавить STATE_CHANGE по модели.

4.

Добавить DETECTOR_TICK агрегировано (len/hash/stop/send/hidden/focus) + DETECT_DONE.

5.

SELECTOR_MISS агрегировать: счётчик + 1–2 примера, плюс HIT rate.

6.

Structured errors: PORT_CLOSED, HANDSHAKE_TIMEOUT, RECOVERY_ACTION.

7.

Integrity snapshots при MODEL_MISSING и перед cleanup.

4) Что можно считать “бесполезным” в текущем виде и убрать/переформатировать

-

Текстовые label шагов (“Typing start…”) → перевести в PIPELINE_STEP.

-

Дубли PIPELINE_COMPLETE → оставить один canonical event.

-

sampled/sampleRate в debug-режиме → либо убрать, либо сделать единым на уровне файла/сессии (не в каждом событии).

-

Поэлементный SELECTOR_MISS без HIT → заменить на агрегаты по окну времени.

-

и предложу минимальный план миграции, чтобы не ломать текущие экспорты All Logs/JSON.

Анализ телеметрии и логов в контексте `tab-manager.js` показывает классическую проблему "шума", за которым теряются критические сигналы о состоянии браузера.
### 1. Что бесполезно (Шум)
1.  **Дублирование событий `TAB_VISIT`:**    *   *Наблюдение:* В логах каждое переключение вкладки записывается дважды: один раз как тип `VISIT`, второй раз как `TELEMETRY`.    *   *Пример:* `[21:44:40] TAB_VISIT — 7534ms` (x2).    *   *Решение:* Оставить только одно событие. Это сократит объем логов на 10-15%.
2.  **Избыточность Round 0 (Open Tabs):**    *   *Наблюдение:* Для каждой модели генерируется 4 события: `ROUND0_START`, `opening tab`, `TAB_CREATED` (или `ROUND0_TAB_OPENED`), `ROUND0_END`.    *   *Пример:* Для 8 моделей это 32 записи, просто чтобы сказать "вкладки открыты".    *   *Решение:* Оставить только событие `TAB_CREATED` (или `ATTACHED`) и одно суммарное событие `ROUND0_COMPLETE`.
3.  **Спам `SELECTOR_MISS`:**    *   *Наблюдение:* Claude и Perplexity генерируют сотни записей `SELECTOR_MISS` в секунду, когда вкладка в фоне (DOM не рендерится).    *   *Вред:* Это забивает буфер логов и вытесняет полезную информацию о том, *почему* селектор не найден (например, статус загрузки вкладки).    *   *Решение:* Агрегировать эти ошибки на стороне `tab-manager` или content-script (например, "Selector X missed 50 times over 5s").
### 2. Чего не хватает (Слепые зоны)
1.  **Статус `discarded` (Выгрузка из памяти):**    *   *Проблема:* Мы видим тайм-ауты, но не знаем, была ли вкладка физически выгружена Chrome для экономии памяти (`tab.discarded`).    *   *Нужно:* В `ensureTabReadyForDispatch` логировать не просто `ok/fail`, а явный статус: `active`, `idle`, `discarded`, `unloaded`.
2.  **Статус Audio Context (Keep-Alive):**    *   *Проблема:* Мы внедряем Audio Hack, но `tab-manager` не знает, сработал ли он.    *   *Нужно:* Логировать факт успешного запуска AudioContext во вкладке. Если вкладка говорит "Ready", но AudioContext не запущен — это кандидат на зависание.
3.  **Привязка к окну (`windowId`):**    *   *Проблема:* Если пользователь вытащил вкладку в отдельное окно, логика фокуса может вести себя иначе.    *   *Нужно:* Логировать `windowId` при создании и проверке, чтобы понимать, все ли вкладки находятся в одном контейнере.
4.  **Время с последнего доступа (`lastAccessed`):**    *   *Проблема:* Chrome агрессивно троттлит вкладки, которые не открывались более 5 минут.    *   *Нужно:* При проверке готовности логировать `Date.now() - tab.lastAccessed`, чтобы видеть корреляцию между "давностью" вкладки и ошибками тайм-аута.

INFO

WARN

ERROR

DEBUG
Отладочная информация (значения переменных, трассировка).

Новая архитектура телеметрии:
javascript

// telemetry-service.js
class TelemetryService {
constructor() {
this.sessionId = generateSessionId();
this.eventChain = new Map();
}

emit(event) {
// 1. Добавить контекст
event.sessionId = this.sessionId;
event.chainId = this.buildChainId(event);
event.parentId = this.getLastEventId();

// 2. Валидация и фильтрация
if (this.isDuplicate(event)) return;
if (this.isStale(event)) return;

// 3. Сохранение в цепочку
this.eventChain.set(event.id, event);

// 4. Отправка (batch)
this.batchQueue.push(event);
if (this.batchQueue.length >= 10) {
this.sendBatch();
}

// 5. Локальное хранение для отладки
this.saveToLocalStorage(event);
}

// Интеллектуальные метрики
calculateActivationPriority(model) {
return {
model,
score: this.computeScore([
this.getTimeSinceLastActivation(model),
this.getDOMActivity(model),
this.getAnswerProbability(model)
]),
recommendedActions: this.suggestActions(model)
};
}
}

Humanoid Effectiveness = успешные действия / все действия

Answer Detection Rate = обнаруженные ответы / ожидаемые ответы

Tab Activation Latency = время от запроса до активации

Round Completion Ratio = завершённые раунды / все раунды
